## 五、闭环缺口 · 旗舰案例：updater「看似完整实则整条链断裂」

`src-tauri/src/updater.rs` 是本次最典型的"写了代码 ≠ 功能可用"反面教材，**四层同时断裂**：

1. **模块层**：`lib.rs` 顶部**没有 `mod updater;`**（grep 确认无匹配）→ 整个文件是 git 未跟踪的**孤儿，根本不参与编译**。里面 `check_update`/`get_app_version`/`open_release_page`/`download_update`/`launch_installer_and_quit` 五个命令实现体（含 semver 比较、msi 校验、单元测试）再漂亮也无效。
2. **注册层**：未进 `generate_handler!`、未进 `build.rs` COMMANDS、未进 capabilities → 即便编译，invoke 也会 command-not-found。
3. **前端封装层**：`api.ts` **根本没导出** getAppVersion/checkUpdate/openReleasePage，而 `settings.ts:266/279/293` 调用之 → 运行时先抛 **TypeError**（连 invoke 都到不了）。
4. **爆炸半径**：
   - `settings.ts:265-268` 在 `renderSettings` 主体内**同步**调用 getAppVersion，异常上抛使 `root.appendChild`（settings.ts:323）永不执行 → usage/theme/advanced/proxy/about 全部不渲染 = **整个设置面板空白**。
   - `npm run build`（`tsc && vite build`）因 TS2339 **在打包前直接失败**——**实测 `npx tsc --noEmit` 退出码 2**，报错 4 处（settings.ts 266/267/279/293），属发布硬阻断。

用户观感：进设置 → 整页打不开；即便绕过，"检查更新"恒走 catch 显示"检查失败，请检查网络后重试"（把"命令不存在"伪装成"网络问题"，二次误导）；文案写 GitHub 实为 Gitee，源都对不上。**整套自动更新对用户价值为零。**

**处置建议（二选一）：**
- **接通**：`lib.rs` 加 `mod updater;` → 5 命令进 generate_handler! + build.rs COMMANDS + capabilities/default.json → `api.ts` 补导出封装 → 修正 GitHub/Gitee 文案 → 更新源改为可配置。
- **移除**：删 `updater.rs` 与 `settings.ts:262-321` 整个关于/检查更新区块，消除死链接与构建失败。

其余闭环状态：49 个已注册命令四处闭环成立、实现体真实、字段对齐。`pick_app`、`set_settings_persist` 属"已注册但前端无 invoke"的死命令（用户看不到，非欺骗，建议清理）。
