# iTools 开发准则

> 本准则的第一目标：**杜绝夸大功能与虚假实现**。任何人（包括 AI 助手）参与 iTools 开发，都必须遵守；每次功能完成、每次发布前，都要按本准则的「审查清单」核对。

---

## 一、诚信实现准则（红线，不得违反）

1. **真实实现优先**
   每个声称存在的功能，必须有真实、可运行的实现。**禁止**用空桩、写死的假返回、硬编码假数据来「冒充」已实现。命令不能只注册不干活。

2. **UI 与实现必须一致（不夸大）**
   界面上的每一个按钮、开关、输入框、菜单项、文案，其行为必须与后端实际能力一致。**禁止**出现「看起来能用、点了却不生效」的控件（例如：能打开、能保存，但后端根本不读取这个值）。

3. **未完成必须对用户诚实标注**
   占位 / 桩 / 演示 / 仅本地 / 未接后端 的功能，必须在 **UI 上明确告知用户**（如「敬请期待」「演示」「本地资料」「未生效」），或直接禁用 / 隐藏。**不得**让用户误以为可用。「敬请期待」式占位是诚实的；「假装能用」是欺骗。

4. **新增命令必须四处闭环**
   新增 Tauri 命令，必须同时满足：① 有真实 `#[tauri::command]` 实现（非空桩）；② 在 `lib.rs` 的 `generate_handler!` 注册；③ 在 `build.rs` 的 `COMMANDS` 白名单；④ 前端 `invoke` 的命令名 / 参数 / 返回字段与后端精确一致。缺任何一环都算未完成。

5. **前后端接口一致（防幻觉）**
   前端 `invoke("命令名", {参数})` 与后端函数名 / 参数名必须对得上；返回结构体字段（含 `serde(rename_all)` 后的实际名）与前端 interface 及所有字段访问必须一致。**禁止**臆造不存在的命令、参数或字段。

6. **不谎报，如实汇报**
   交付或汇报时，必须如实区分：哪些是**真实现**、哪些是**桩 / 占位**、哪些**未验证**。测试失败、步骤跳过、仅编译未真机验证，都要明说。**禁止**把「写了代码」说成「功能可用」。

7. **联网类功能特别约束**
   账户、登录、数据同步、云、代理等「看起来要联网」的功能，在**真正接入后端之前**，必须在 UI 标注真实状态（如「本地资料（未接入云账号）」「代理未生效」），**禁止**用本地桩伪装成联网能力。措辞上避免「注销账号」「云同步」等暗示服务端却不兑现的词。

8. **安全与隐私红线**（详见 README「开发约定」）
   - 不硬编码个人 / 敏感数据（昵称、路径、邮箱等）；账号类默认值必须为空。
   - 凭据（token / 密钥 / 密码）只从**环境变量**读取，源码与二进制零明文。
   - 服务端地址 / 域名 / endpoint 必须可配置，不写死进代码。
   - Release 版不产调试日志。

## 二、发布前审查清单（每次功能完成 / 发布前逐项核对）

- [ ] 每个 UI 控件都能追到**真实生效**的后端实现（UI → invoke → 命令实现 全链路走通）？
- [ ] 有没有「存了不生效」的开关 / 按钮 / 输入（改了没有任何实际效果）？
- [ ] 所有占位 / 桩 / 仅本地 功能，都已在 UI **明确标注**，不会误导用户？
- [ ] 新增命令是否 **实现 + `lib.rs` 注册 + `build.rs` 白名单 + 前端调用** 四处闭环？
- [ ] 前后端命令名 / 参数 / 返回字段是否**逐一对得上**（无幻觉、无拼写漂移）？
- [ ] 有没有**硬编码假数据**冒充真实结果？
- [ ] 联网类功能是否**如实反映**「已接入 / 未接入」？
- [ ] 是否有硬编码的个人数据 / 本机路径 / 明文凭据 / 写死的服务端地址？
- [ ] `cargo test` / `cargo clippy`（0 warning）/ 前端 `tsc` 是否通过？关键功能是否**真机验证**过？
- [ ] 汇报是否如实分清了「真实现 / 桩 / 未验证」？

---

## 三、审查机制（怎么执行）

**何时审**：每次功能实现完成后、以及每次发布前，必须执行一次。

**怎么审**：
- **追全链路**：从用户能看到的 UI 入口，一路追到后端实现体，确认真实生效——不要凭 UI 外观或命令名就判定「已实现」。
- **对抗性视角**：默认怀疑「这个功能是不是假的 / 夸大的」，主动去找空桩、假返回、不生效的控件、对不上的接口。有条件时用**多个独立视角**分别审再交叉验证（例如按功能模块并行审查 + 对抗验证），减少单一视角的盲点。
- **四档判定**：对每个功能给出结论——
  - `真实`：有真实现、确实生效；
  - `诚实占位`：未完成但已如实标注（可接受）；
  - `欺骗性桩`：UI 像能用、实际不生效且未告知用户（**必须修**）；
  - `虚假`：声称有、后端根本没有或返回假数据（**必须修**）。
- 判为「欺骗性桩」或「虚假」的，发布前必须修复：要么补全真实现，要么如实标注 / 禁用 / 移除。

**记录**：审查结论（尤其欺骗性 / 虚假项）应向需求方如实呈现，不得隐瞒或美化。
